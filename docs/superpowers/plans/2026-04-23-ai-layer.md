# AI Layer Plan

## Goal

Ship the first AI-assisted layer on top of the working procurement workflow:

- convert buyer free-text briefs into structured requirement drafts
- generate evidence-grounded shortlist rationale
- explain quote differences and assumptions in plain language
- assist ops and buyers without replacing explicit human decisions

This phase should end with AI features that make the existing workflow faster and clearer while staying grounded in structured product data.

## Product outcome

At the end of this phase, the team can:

- accept a buyer's natural-language sourcing brief and map it into the requirement schema
- suggest missing fields or ambiguities before RFQ issuance
- explain why vendors were shortlisted or excluded using stored rules and data
- summarize quote differences, assumptions, and anomalies from normalized quote records
- provide internal assistive UX without changing the system of record

This phase improves workflow speed and comprehension. It should not become the product's only reason to exist.

## Scope

### In scope

- LLM-assisted requirement parsing into existing schema
- field confidence and ambiguity detection
- shortlist rationale generation grounded in vendor and requirement data
- quote comparison explanations grounded in structured quote data
- prompt and response logging for QA
- evaluation datasets and offline acceptance tests
- safety rules for grounded responses

### Out of scope

- autonomous RFQ issuance
- autonomous vendor selection
- chat-first product redesign
- external agent access or MCP server
- scraped public evidence ingestion
- generic cross-category intelligence

## Phase objective

Prove that AI can materially reduce buyer and ops effort without weakening trust, auditability, or workflow control.

The right bar here is not "sounds smart." The right bar is:

- faster requirement creation
- fewer malformed RFQs
- better buyer comprehension of quote differences
- no hidden or fabricated reasoning

## Design principles

### AI is assistive, not authoritative

The model can suggest, extract, summarize, and explain. It does not finalize workflows or mutate core records without explicit user confirmation.

### Structured data stays primary

AI reads from and writes to the existing typed schema. Do not introduce a parallel AI-only data model.

### Every explanation must be grounded

If the system cannot cite the underlying requirement fields, vendor records, compliance records, or quote line items, it should not present the claim as fact.

### Confidence must be visible

For extraction tasks, show confidence and unresolved ambiguities instead of pretending certainty.

## AI surfaces

### Buyer requirement intake assistant

Input:

- a buyer's natural-language brief
- optional structured form fields already entered

Output:

- draft values for the requirement schema
- missing required fields
- ambiguous fields needing confirmation
- normalized plain-language summary for review

### Shortlist rationale assistant

Input:

- requirement record
- shortlisted vendors
- exclusion reasons

Output:

- concise rationale per shortlisted vendor
- concise exclusion rationale for filtered-out vendors
- warnings where shortlist quality is weak due to low supply

### Quote explanation assistant

Input:

- normalized quotes
- line items
- assumptions
- vendor trust and verification metadata

Output:

- summary of key differences across vendors
- explanation of cost drivers
- flagging of assumption mismatches
- plain-language compare summary for buyer review

### Internal ops assistance

Input:

- vendor records
- checklist items
- compliance metadata

Output:

- optional summaries of missing information
- draft review notes for ops

Keep ops assistance secondary in this phase. Buyer-side workflow help is the main priority.

## System architecture

### Core components

1. `ai_tasks` table for request/response lifecycle
2. `ai_evaluations` table for benchmark runs
3. prompt builders for each AI use case
4. grounding adapters that fetch structured context
5. response validators that map outputs into typed schema
6. UI review components that require user confirmation before writeback

### Suggested service boundaries

- `src/server/services/ai/requirement-parsing/...`
- `src/server/services/ai/shortlist-rationale/...`
- `src/server/services/ai/quote-explanations/...`
- `src/server/services/ai/shared/grounding.ts`
- `src/server/services/ai/shared/validation.ts`
- `src/server/services/ai/shared/evals.ts`

## Requirement parsing design

This is the highest-value AI surface in Phase 4.

### Input contract

- free-text brief
- buyer org context
- category context
- allowed enum values and schema hints

### Output contract

- parsed structured fields
- `missing_fields`
- `ambiguous_fields`
- `confidence_by_field`
- `normalized_summary`

### Example fields to parse

- city
- site type
- headcount
- shift pattern
- day/night coverage
- relief requirement
- supervisor requirement
- contract term
- start date
- compliance requirements
- response deadline if mentioned

### UX flow

1. Buyer pastes or types a free-text requirement.
2. System generates a draft structured form.
3. Buyer reviews suggestions field by field.
4. Buyer confirms or edits.
5. Only then is the requirement record saved or updated.

## Shortlist rationale design

This feature should explain deterministic shortlist results, not invent hidden scoring.

### Grounding inputs

- requirement fields
- `vendor_shortlist_snapshots`
- vendor verification state
- compliance completeness
- service area match

### Output shape

- `recommended_reason`
- `watchouts`
- `exclusion_reason`

### Guardrails

- never state unverified facts about vendor quality
- never imply guaranteed performance
- cite only fields already stored in the system

## Quote explanation design

This is the second high-value AI surface.

### Grounding inputs

- latest submitted quotes for compared vendors
- quote line items
- assumptions JSON
- vendor verification metadata

### Output shape

- top-level summary
- cost driver bullets
- assumption mismatch bullets
- anomaly flags
- suggested buyer questions

### Guardrails

- do not recommend a winner automatically
- do not flatten distinct assumptions into "best price"
- call out missing data explicitly

## Model behavior rules

### Requirement parsing

- prefer extraction over creativity
- return `null` for missing values
- use enums exactly as defined by product schema

### Explanations

- stay concise and operational
- attach citations to each major claim
- distinguish facts from inferences

### Failure behavior

- if confidence is low, return missing/ambiguous fields rather than guessing
- if grounding data is incomplete, respond with a partial explanation and explicit gaps

## UI requirements

### Buyer intake

- free-text input box
- structured form preview
- highlighted uncertain fields
- accept/edit controls before save

### Shortlist page

- "why this vendor" explanation block
- "why excluded" detail for filtered vendors

### Compare page

- AI summary panel
- expandable citations
- warnings for low-confidence or incomplete explanations

## Evaluation plan

Do not ship these features on vibes. Build a small but real eval set.

### Requirement parsing eval set

- 25 to 50 anonymized buyer briefs
- expected structured outputs
- expected ambiguity flags

### Quote explanation eval set

- 15 to 20 real or synthetic normalized quote sets
- expected cost-driver explanations
- expected mismatch detection

### Acceptance thresholds

- high precision on required field extraction
- low hallucination rate on explanations
- explicit ambiguity detection preferred over wrong confident extraction

## Testing strategy

### Unit tests

- output schema validation
- citation attachment logic
- requirement field mapping
- confidence threshold behavior

### Integration tests

1. Buyer submits free-text brief and receives structured draft.
2. Buyer confirms draft and requirement saves correctly.
3. Shortlist rationale is generated from stored shortlist data.
4. Quote explanation references actual quote records and line items.
5. Low-confidence output triggers review warnings instead of blind writeback.

### Manual QA scenarios

1. Ambiguous brief with missing headcount and start date.
2. Brief containing mixed structured and free-text details.
3. Compare set where cheapest quote has weaker assumptions.
4. Compare set with missing line items in one vendor response.
5. Low-supply scenario where shortlist quality is weak.

## Important implementation notes

### Do not hide the structured workflow

The UI should still expose the underlying form and compare logic. AI is there to accelerate and explain, not replace product clarity.

### Keep prompts schema-bound

Prompt contracts should mirror the existing requirement and quote schema closely. Loose prompts will create expensive cleanup work.

### Store enough for QA, but be careful with sensitive data

Log AI inputs and outputs for evaluation, but minimize raw personal or confidential text where possible.

### Separate fact from inference

A good pattern is:

- facts from system data
- inferences labeled as likely or possible
- questions when the system cannot conclude safely

## Risks

### Risk: hallucinated explanations weaken trust

Mitigation:

- grounding adapters
- citations
- explicit missing-data handling
- offline evals before rollout

### Risk: requirement parsing feels magical but inaccurate

Mitigation:

- reviewed draft UX
- field-level confidence
- ambiguity prompts instead of forced guesses

### Risk: AI complexity slows core roadmap

Mitigation:

- build only on top of proven flows
- keep each AI surface independent and feature-flagged

### Risk: buyers over-trust generated summaries

Mitigation:

- always show underlying structured data
- avoid auto-recommending winners
- keep manual decision capture explicit

## Deliverables

1. AI task logging and evaluation infrastructure
2. Requirement parsing assistant
3. Shortlist rationale assistant
4. Quote explanation assistant
5. Feature flags and fallback UX

## Definition of done

This phase is done when:

- buyers can create structured requirement drafts from free text with human confirmation
- shortlist and quote explanations are grounded in stored system data
- AI outputs are measurable through evals and logs
- low-confidence or unsupported outputs fail safely
- the product is more useful with AI, but still fully usable without it

## Recommendation after this phase

Build Phase 5 next: scraped ingestion plus public vendor pages.

Reason:

- by then the core procurement loop and assistive layer are validated
- scraped records can be added as supply acquisition without contaminating the trust surface
- public pages and later MCP access can sit on top of the curated data model cleanly
